# Feature Specification: 桌面主Agent可靠性修复与测试框架

**Feature Branch**: `[001-desktop-agent-reliability]`  
**Created**: 2026-05-07  
**Status**: Verified
**Input**: User description: "修复星星的vibecoding启动器桌面主流程中的命名错误、发送卡死、过早出方案、选方案后不落盘问题；把 Speckit、tmp 测试数据、docs 真源文档和自动测试框架纳入正式开发约束。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 稳定和星星自然对话 (Priority: P1)

作为使用 `星星的vibecoding启动器` 的用户，我希望进入对话阶段后看到的是 `星星` 的人格与按钮文案，并且每次发送消息时界面明确显示处理中状态，而不是让桌面窗口看起来像“未响应”。

**Why this priority**: 这是桌面主路径最先暴露给用户的交互面。如果发送时卡死或命名错误，用户会立即失去信任，后续流程无法继续验证。

**Independent Test**: 启动桌面主流程，进入对话阶段，确认标题、消息发送按钮、占位提示均以 `星星` 为主体；点击发送后立即出现加载态，窗口可继续响应，直到收到返回结果。

**Acceptance Scenarios**:

1. **Given** 用户已完成渠道与工作区校验，**When** 进入对话阶段，**Then** 页面标题与按钮文案显示为 `星星` 与 `发送给星星`，不得再出现 `主Agent 对话` 或 `发送给主Agent`。
2. **Given** 用户已输入对话内容，**When** 点击发送，**Then** 页面立即显示加载反馈并禁止重复提交，桌面窗口保持可响应。

---

### User Story 2 - 在真正理解需求后再出方案 (Priority: P1)

作为用户，我希望 `星星` 在真正理解产品目标后再输出三个解决方案，而不是仅凭固定轮次阈值过早结束需求澄清。

**Why this priority**: 过早出方案会让整个“LLM 驱动”沦为假对话，直接破坏产品核心定位。

**Independent Test**: 用测试夹具发送多轮信息不足的对话，验证系统只会继续追问；只有在关键信息齐备或用户确认 `星星` 总结准确后，才允许进入三方案阶段。

**Acceptance Scenarios**:

1. **Given** 用户只给出产品名称或模糊想法，**When** 连续发送 2-3 轮信息仍不完整，**Then** `星星` 必须继续追问，不得仅因轮次达到阈值就输出方案。
2. **Given** 用户已经明确产品形态、目标用户、核心问题、关键功能和约束，**When** `星星` 完成总结并获得确认，**Then** 系统才可以输出三个完整方案。

---

### User Story 3 - 选定方案后必须生成真实初始化结果 (Priority: P1)

作为用户，我希望在选择最终方案后，目标工作区会真正生成初始化产物，而不是只显示“成功”消息。

**Why this priority**: 这是当前最严重的假成功问题。没有真实落盘，就不能称为初始化完成。

**Independent Test**: 在临时工作区里完成一次完整桌面流程，选择任一方案后，验证工作区生成初始化文档、协议文件和 session 产物，并能通过对应 postcheck。

**Acceptance Scenarios**:

1. **Given** 用户在方案弹窗中确认一个方案，**When** 初始化结束，**Then** 目标工作区必须生成至少一套与兼容 bootstrap 等价的初始化产物，而不是空目录。
2. **Given** 生成阶段失败或 postcheck 未通过，**When** 桌面收口，**Then** 系统必须阻断成功提示并给出失败原因。

---

### User Story 4 - 有内建测试框架而不是依赖人工兜底 (Priority: P2)

作为维护者，我希望仓库内有明确的自动测试框架、`tmp/` 测试数据约定和 `docs/` 真源文档约定，这样每次修复桌面主流程时都能先在本地自测，再交给最终人工验收。

**Why this priority**: 没有稳定的自测框架，桌面端每次改动都会重新回到“肉眼点点看”的高风险状态。

**Independent Test**: 执行统一测试入口，自动创建和清理 `tmp/` 下的测试工作区，完成对话、方案生成、落盘、文档检查和回归校验，并输出明确结果。

**Acceptance Scenarios**:

1. **Given** 仓库维护者执行自动测试入口，**When** 测试开始，**Then** 所有临时测试数据都必须写入 `tmp/`，不得污染正式 docs 或业务目录。
2. **Given** 维护者查阅项目文档，**When** 查看真源文档，**Then** 能明确知道 `docs/` 存放正式文档，`tmp/` 只存测试夹具、测试输出和临时数据。

### Edge Cases

- 当远程 LLM 接口响应很慢时，桌面是否仍然保持可响应并给出加载态？
- 当 `星星` 的总结尚未被用户确认时，系统如何阻止误入三方案阶段？
- 当方案确认后 bootstrap 生成失败、postcheck 失败或目标目录无写权限时，系统如何阻断假成功？
- 当自动测试重复运行时，`tmp/` 下旧的测试夹具和输出如何安全隔离与清理？
- 当 Windows 环境没有 `bash/sh` 时，Speckit 计划文件如何仍然作为正式开发入口存在？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 桌面对话阶段所有用户可见命名都 MUST 以 `星星` 为主体，包含阶段标题、消息发送按钮、占位提示和方案弹窗说明。
- **FR-002**: 用户点击发送消息后，前端 MUST 立即进入可见加载态，并在请求完成前禁用重复提交。
- **FR-003**: 后端对话处理 MUST 采用不会阻塞桌面窗口事件循环的执行方式。
- **FR-004**: 方案输出条件 MUST 基于“需求信息完整度与用户确认状态”，不得仅基于固定轮次阈值。
- **FR-005**: 系统 MUST 能跟踪至少以下关键信息槽位：产品形态、目标用户、核心问题、关键功能、技术或交付约束、用户是否确认当前理解。
- **FR-006**: 用户确认方案后，桌面流程 MUST 触发真实初始化落盘链路，在目标工作区生成项目协议、文档和 session 产物。
- **FR-007**: 初始化落盘结果 MUST 通过等价的 postcheck 或明确失败阻断，不得在工作区为空时显示成功。
- **FR-008**: 仓库级 `AGENTS.md` MUST 明确要求后续线程优先使用 `Speckit` 制定计划，并把 `Speckit`、`GitNexus`、`superpowers`、`agent-browser`、`chrome-devtools`、`Speckit` 视为能力约束的一部分。
- **FR-009**: 仓库 MUST 提供自动测试框架，用于覆盖对话加载态、过早出方案阻断、方案确认后落盘和文档约束校验。
- **FR-010**: 自动测试框架产生的测试数据、临时工作区、模拟响应和快照 MUST 写入 `tmp/`，不得混入 `docs/` 或正式产物目录。
- **FR-011**: 真源文档 MUST 明确 `docs/` 仅存放项目正式文档，`tmp/` 仅存放测试夹具、测试输出与临时数据。
- **FR-012**: 测试框架 MUST 在交付前由系统自行运行并输出通过/失败结果，不能把最终有效性验证完全留给人工兜底。

### Key Entities *(include if feature involves data)*

- **ConversationReadinessState**: 表示当前桌面对话是否已经收集到足够信息进入方案阶段，包含必填槽位、缺失项、用户确认状态和最近一次总结。
- **DesktopSolutionSelection**: 表示用户在桌面端选定的方案，包含方案 ID、方案摘要、是否已确认、是否已触发 bootstrap、是否已完成 postcheck。
- **DesktopBootstrapResult**: 表示桌面主流程在目标工作区的真实初始化结果，包含目标路径、生成文件摘要、postcheck 状态、失败原因和 handoff 路径。
- **DesktopTestFixture**: 表示 `tmp/` 下的自动测试夹具，包含临时工作区、模拟 provider 响应、期望日志和清理策略。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户在对话阶段的所有主要交互文案中，`星星` 命名覆盖率达到 100%，不再出现 `主Agent` 作为对外主称谓。
- **SC-002**: 在模拟 3 秒以上远程响应的情况下，用户点击发送后的 200 毫秒内即可看到加载反馈，且桌面窗口保持可交互。
- **SC-003**: 在信息不完整的测试对话中，系统 100% 保持澄清模式；只有满足信息完整度和确认条件的测试对话才会进入三方案阶段。
- **SC-004**: 在自动测试工作区中确认方案后，100% 生成初始化产物；如果生成失败或 postcheck 未通过，100% 阻断成功提示。
- **SC-005**: 交付前执行的自动测试框架能够一键完成核心回归，并将测试夹具与输出全部限定在 `tmp/` 下。

## Assumptions

- 现有兼容初始化编排器仍然是最可靠的文档生成与 postcheck 能力来源，桌面主路径将优先复用它，而不是新写一套不兼容生成器。
- `星星` 的桌面对话不要求在本轮成为完全开放式长期记忆 Agent，但必须至少具备基于信息完整度的多轮澄清能力。
- 本轮允许为 Windows 环境补 Speckit 计划与测试辅助层，但不会把整个项目强制改造成必须依赖 WSL 才能开发。
- 自动测试框架的第一优先级是覆盖当前桌面主路径的阻断问题，不追求一次性覆盖所有未来扩展场景。
