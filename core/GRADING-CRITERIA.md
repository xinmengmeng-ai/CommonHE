# Grading Criteria

## 目标

定义评估者在迭代评估和最终门禁中使用的通用评分维度、阈值与校准指引。将主观质量判断转化为可分项评分的具体维度。

## 通用维度（所有任务）

### 功能完整性（Functionality）

| 分数 | 含义 |
|------|------|
| 9-10 | 所有功能正常，边界情况完善 |
| 7-8 | 核心功能正常，少量边界未覆盖 |
| 5-6 | 主要功能可用，有明显缺失 |
| 3-4 | 核心功能部分工作，多处异常 |
| 1-2 | 基本不可用 |

**默认阈值：>= 7**

### 代码质量（Code Quality）

| 分数 | 含义 |
|------|------|
| 9-10 | 结构清晰，命名精准，职责分明 |
| 7-8 | 结构合理，少量可改进点 |
| 5-6 | 可工作但有明显技术债 |
| 3-4 | 结构混乱，难以维护 |
| 1-2 | 无结构可言 |

**默认阈值：>= 6**

### 规格合规（Spec Compliance）

| 分数 | 含义 |
|------|------|
| 9-10 | 完全符合 contract，无偏离 |
| 7-8 | 基本符合，少量合理偏离 |
| 5-6 | 多处偏离 contract |
| 3-4 | 严重偏离或大量范围蠕变 |
| 1-2 | 与 contract 几乎无关 |

**默认阈值：>= 7**

### 测试覆盖（Test Coverage）

| 分数 | 含义 |
|------|------|
| 9-10 | 核心路径和边界均有测试 |
| 7-8 | 核心路径有测试，边界部分覆盖 |
| 5-6 | 有基本测试但覆盖不足 |
| 3-4 | 测试极少或不可运行 |
| 1-2 | 无测试 |

**默认阈值：>= 6**

## 可选扩展维度

项目可按需追加以下维度：

### 前端扩展
- 设计质量（Design Quality）：视觉连贯性与身份感
- 原创性与工艺（Originality & Craft）：是否有自定义设计决策
- 响应式与可访问性（Responsiveness & Accessibility）

### 后端扩展
- API 契约合规（API Contract Compliance）
- 性能（Performance）
- 安全性（Security）
- 错误处理（Error Handling）

### 项目特定扩展
各项目可根据自身需求定义专属维度（如多租户合规、信创兼容等），遵循同样的评分量表和阈值规则。

## 评分规则

### 硬性阈值
- 任何单一维度低于其阈值 -> 整体 `fail`
- 所有维度达到阈值 -> 整体 `pass`

### 权重分配
- **高权重**：功能完整性、规格合规、安全性
- **中权重**：代码质量、测试覆盖、API 契约合规
- **低权重**：设计质量、原创性、响应式、性能

高权重维度的 `fail` 必须在下一轮迭代中优先修复。

### 维度裁剪
Sprint Contract 中应约定本次任务启用哪些维度。不相关的维度应显式标注"本次不评分"。

## 校准样例格式

为帮助评估者校准评分尺度，建议为每个关键维度提供至少 2 个校准样例：

```markdown
### 样例：<维度名> = <分数>（<pass/fail>）

> 场景：<任务描述>
> 实际：<实现结果描述>
> 判定：<评分依据> -> <分数>
```

项目应根据实际经验持续积累校准样例，提升评估一致性。
