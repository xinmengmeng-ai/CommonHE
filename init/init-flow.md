# Init Flow

## 总体流程

`星星的vibecoding启动器` 的桌面主流程必须按以下顺序工作：

1. 启动 Windows portable exe
2. 进入本地环境准备阶段
3. 选择正式渠道
4. 读取渠道配置来源与可用模型
5. 完成鉴权与轻量连通性校验
6. 选择目标 workspace
7. 创建主 Agent 会话
8. 主 Agent 以自然语言方式理解用户目标
9. 主 Agent 结合 `agency-agents-zh/README.md` 推断团队组成
10. 主 Agent 输出三个解决方案
11. 程序调用内置解决方案选择弹出框
12. 用户完成选择
13. 用户填写初始化协作包项目名
14. 用户选择后续接管软件（v1.0：Codex；v2.0 规划：Claude Code）
15. 用户确认必需安装或配置的 Skill / MCP / CLI 能力
16. 程序生成目标软件原生入口的初始化协作包
17. 初始化语境正式收口

桌面主流程不再以旧的固定 `discovery / proposal / confirm` 作为标准路径。

---

## 阶段说明

### 阶段 1：环境准备

要求：

- 产品主名称必须显示为 `星星的vibecoding启动器`
- Windows portable exe 是首选入口
- Desktop 基于 Tauri
- 渠道准备先于 workspace 选择
- 当前第一批正式可用渠道为：
  - `DeepSeek`
  - `Custom`
  - `Codex`
- `Claude Code` 与 `Gemini CLI` 当前可保留检测信息，但不属于第一批正式可用桌面主流程渠道

### 阶段 2：渠道与模型确认

要求：

- 读取渠道配置后，必须展示真实模型列表或显式要求先读取模型列表
- 默认禁止用户自由输入模型名称
- 只有显式选择“自定义模型”时，才允许手工输入
- `config detected` 不等于 `provider usable`
- `provider usable` 不等于 `model usable`
- 必须明确区分：
  - 配置已发现
  - 模型已发现
  - 鉴权已验证
  - 渠道可调用

### 阶段 3：正式渠道规则

#### DeepSeek

- 必须作为正式内置渠道存在
- 默认模型为 `deepseek-v4-flash`
- 可选模型为 `deepseek-v4-pro`
- APIKey 为空时必须阻断

#### Codex

- 当前桌面第一版按 API 渠道支持
- 不做本地 CLI 会话桥接
- 若检测到本地配置来源，只能作为“可解析配置线索”，不能直接冒充“已可用”
- 必须形成真实 API 请求闭环后，才允许进入主 Agent 会话

#### Custom

- 必须视为正式远程渠道，而不是兜底输入框
- 必须校验：
  - 必填字段完整性
  - URL / 协议合法性
  - 轻量 API 连通性
- 若可行，应优先读取远程模型列表，再由用户选择模型
- 只有用户显式选择“自定义模型”时，才允许自由输入模型名

### 阶段 4：workspace 选择

要求：

- 只有在渠道与模型环境通过验证后，才允许继续选择 workspace
- workspace 只是主 Agent 的上下文边界，不再自动触发旧 discovery/proposal 问卷

### 阶段 5：主 Agent 会话

要求：

- 桌面主流程必须由真实主 Agent 接管
- 主 Agent 必须参考：
  - `product-manager.md`
  - `agency-agents-zh/README.md`
- 主 Agent 必须主动询问用户要做的产品形态，例如：
  - MCP
  - Skill
  - 网站
  - 软件
  - 其他产品
- 主 Agent 不得表现成固定问卷
- 程序不得再用固定 discovery 模板冒充主 Agent

### 阶段 6：方案输出

要求：

- 主 Agent 必须输出三个明确解决方案
- 每个方案必须包含：
  - 架构
  - agent 团队组成
  - token 预估
- 程序负责接收方案结果并展示为可选择的 UI

### 阶段 7：方案选择与收口

要求：

- 必须通过内置解决方案选择弹出框完成选择
- 用户完成选择后，必须继续确认初始化协作包项目名
- 用户必须选择后续接管软件；v1.0 正式发版仅承诺 `Codex`
- 用户必须确认必需安装或配置的 Skill / MCP / CLI 能力；`superpowers`、`agent-browser`、`chrome-devtools`、`GitNexus`、`Speckit` 五项当前不可取消
- `Codex` 方向生成 `AGENTS.md` 与 `.codex/` 入口
- `Claude Code` 方向进入 v2.0 规划，必须完成专属入口、文案、Speckit 配置、truth-source gate 和发布包门禁后再开放
- 生成结果必须是初始化协作包，不得声称业务项目成品或业务代码已生成
- 结束后不得继续在当前初始化语境内展开业务实现
- 成功提示必须引导用户在新会话中继续实施

## 成功条件

桌面主流程只有在以下条件全部满足时，才可视为成功：

1. 渠道与模型环境真实可用
2. 主 Agent 已进行真实自然语言对话
3. 三个完整方案已经生成
4. 内置方案选择弹出框已经触发
5. 用户已经完成方案选择
6. 用户已经确认项目名、目标软件和建议能力选择
7. 对应目标软件的初始化协作包已经生成并通过 postcheck
8. 初始化语境已经正式收口

---

## 测试与文档边界

- `docs/` 用于保存正式初始化文档、真源协议和长期保留说明
- `tmp/` 用于保存桌面主流程自动测试框架的临时工作区、测试夹具和可清理输出
- 桌面主流程的自动测试必须优先在 `tmp/` 中完成，不能把测试产物混入正式文档目录

---

## 失败条件

以下任一情况都不得宣布初始化成功：

1. 只发现配置，但未验证可用性
2. 模型仍然依赖自由输入
3. `custom` 未通过真实校验
4. `DeepSeek` 空 APIKey 仍然放行
5. 主 Agent 未真正接管对话
6. 仍由固定 discovery/proposal 模板驱动桌面主流程
7. 三个方案未完整输出
8. 用户尚未完成方案选择

---

## 兼容 / 回归入口

当用户不用桌面主流程，而是在项目根目录输入 `初始化 CommonHE` 时，仍允许走兼容路径：

1. 识别宿主入口环境
2. 执行 `doctor`
3. 通过后执行 `precheck`
4. 老项目进入旧 `autodiscovery`
5. 空目录 / 无项目信号目录保留旧 discovery / proposal / confirm
6. 执行 bootstrap
7. 执行 `postcheck`
8. 输出兼容路径成功消息并收口

兼容路径仅用于：

- 历史兼容
- 回归验证
- 无法运行桌面主流程时的补充入口

它不再代表未来产品主方向。
